The Open Model Landscape

Open vs. Closed Models

Open Models

  • Weights publicly available
  • Run locally on your own GPU
  • No per-token API fees
  • Fine-tune and redistribute freely
  • No shutdown blues: a model on your disk can't be deprecated, rug-pulled, or price-hiked

Closed Models

  • Weights hidden behind an API
  • Bound by provider terms & pricing
  • No model portability
  • Convenient but not flexible

Fit Your Hardware

Model Memory ≈ Params × Precision

4×
FP32 bytes per param
2×
FP16 bytes per param
½×
Q4 bytes per param

FP means full-precision numbers; Q4 means weights compressed to 4 bits, about an eighth of the size (quantization gets its own slide soon). A 7B model needs ~28 GB at FP32, ~14 GB at FP16, and ~4 GB at Q4. But these are raw sizes. On system RAM, budget only ~50% for the model.

Two Different Memory Worlds

System RAM is shared with the OS. GPU VRAM is dedicated to compute.

The model is the dominant tenant.

GPU is all but essential for interactive chat. CPU-only inference is seconds-per-token, not tokens-per-second.

Fit Your Hardware

The Unified Memory Exception

Apple Silicon collapses the two memory worlds into one. The win is capacity, not speed.

One pool of memory

CPU and GPU share the same memory on M-series Macs. A 64+ GB Mac loads 70B models that fit on no consumer GPU.

Capacity is the win: models that need a data-centre card elsewhere just load.

The speed trade-off

Decode is bandwidth-bound: 273 GB/s (M4 Pro) to 546 GB/s (M4 Max) vs ~1 TB/s on an RTX 4090. Like-for-like models run 2-3x slower, and prompt processing (compute-bound) is slower still.

Bigger models fit; they don't run faster.

Right-Size with llmfit

Know what your machine can actually run before you download gigabytes. One command; works on macOS, Windows, and Linux.

  • brew install llmfit, or grab a release from github.com/AlexsJones/llmfit
  • Detects your RAM, CPU, and GPU, then scores 200+ models on quality, speed, fit, and context
  • Picks the best quantization for your memory and estimates tokens/sec up front
  • Rule of thumb: run the largest model that stays interactive, not the largest that technically loads

Fit Your Hardware

Model Quantization

Map 32-bit weights onto a smaller discrete set. The single most important technique for consumer hardware.

  • Q4_K_M — ~4.5 GB for 7B params. Best size/quality balance.
  • Q5_K_M — slightly larger, slightly sharper.
  • Q8_0 — nearly FP16 quality. Use if you have VRAM to spare.
  • Code generation tolerates quantization better than math proofs.

What Is Post-Training?

Pre-training gives you a fluent but unaligned base model. Post-training turns it into a useful assistant.

Pre-Training

  • Next-token prediction
  • Massive unlabeled corpora
  • Raw knowledge, no alignment

Post-Training

  • Curated task-specific data
  • SFT + reinforcement learning
  • Instruction following & safety

Post-Training

Supervised Fine-Tuning

Maximize the log-likelihood of high-quality (instruction, response) pairs.

maxθ Σ log Pθ(y | x)
  • Teaches format, style, and shallow patterns
  • Loss computed only on response tokens
  • Necessary first step — gives a sensible starting policy
  • Cannot exceed training data quality
28 / 33
Post-Training

The Need for Reinforcement Learning

Replace fixed labels with a reward signal. The model explores many responses and learns to favour those that score highly.

SFT Mindset

"Here is the correct answer. Copy it."

  • Imitation
  • Single correct path
  • Capped by label quality

RL Mindset

"Try many solutions. Keep what works."

  • Optimization
  • Many paths explored
  • Surpasses label quality

Post-Training

RLVR & GRPO

Reinforcement Learning with Verifiable Rewards. No learned reward model — just ground truth.

RLVR Rewards

Objective, verifiable facts:

  • Unit tests pass → reward = 1
  • Math answer matches → reward = 1
  • Game score achieved
  • JSON validates against schema

GRPO

Group Relative Policy Optimization:

  • No value network needed
  • Sample G responses per prompt
  • Advantage = (rᵢ − r̄) / σᵣ
  • Clipped surrogate + KL penalty

Post-Training

GGUF → Ollama

Closing the loop: export your fine-tuned model and run it locally.

  • Unsloth (the fine-tuning toolkit from Activity F) exports directly to GGUF with your chosen quantization
  • GGUF stores weights, vocab, and metadata in one portable binary
  • Modelfile defines system prompt, temperature, and context length
  • ollama create registers it locally — identical to any Hub model
